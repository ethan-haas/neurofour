# NeuroFour ‚ÄĒ the NeuroGolf metric (exact, mechanical, reproducible)

This file defines scoring. It is part of the build spec (the maker implements it) **and** the contract
the audit checks. Every number here is deterministic given the seed and the registered agents.

## 0. Inputs

- **Position sets.** `bench_data/{train,dev,sealed}.jsonl`, generated by `scripts/gen_positions.py`
  from `NEUROFOUR_SEED` (default `4`). Each line:
  `{"board": "<move-seq or key>", "to_move": 1|2, "value": <-1|0|1>, "optimal_cols": [..], "best_col": c, "scored": {col: score,...}}`.
  Sets are **disjoint** (dedup by canonical key across sets). `train`+`dev` are for learning; `sealed`
  is for strength scoring and must not be used in training. Positions are drawn from self-play/random
  games truncated at varied plies (bias toward non-trivial, non-terminal midgame positions where the
  optimal set is a strict subset of legal moves).
- **Agents.** Everything in `app/agents/registry.py`.

## 1. Strength (higher = better)

Over a position set `P` (default = `sealed`), for each position `p` let `m = agent.select_move(p.board)`:

- **optimality** = `mean[ m ‚ąą p.optimal_cols ]` ‚ÄĒ the headline strength number, in `[0,1]`.
- **blunder_rate** = `mean[ valueafter(p, m) < p.value ]` using the **value-only** convention
  (win=+1, draw=0, loss=‚ąí1), where `valueafter(p,m)` is `‚ąí1 √ó` the value-only solve of the position
  after playing `m` (negamax sign flip). A blunder is any move that *strictly worsens* the achievable
  outcome (e.g. throws away a win, or a draw into a loss). In `[0,1]`, lower is better.
- **soundness** = `1 ‚ąí blunder_rate`.
- Report per-outcome optimality (on winning / drawn / losing positions separately) for diagnostics.

`optimality` and `blunder_rate` are **fully deterministic** for a deterministic agent on a fixed set.
For a stochastic agent, average over `NEUROFOUR_SEED`-seeded repeats (default 1; the flagship must be
deterministic).

## 2. Ladder Elo (secondary strength signal)

Round-robin among all registered agents + fixed reference opponents. Each ordered pair `(A,B)` plays a
fixed roster of **opening books** (the empty board plus a fixed list of short forced-diverse openings)
with `A` first, then `B` first ‚ÄĒ so color is balanced. Win = 1, draw = 0.5, loss = 0. Fit Elo by
logistic MLE (anchor `random` at Elo 0, or iterate the standard update to convergence with a fixed
seed and k-factor). Elo is a *secondary* ranking signal and a UI axis; the gate metric is `optimality`.

## 3. Cost (lower = better)

- **size_bytes** ‚ÄĒ the real on-disk byte length of the agent's decision artifact (weights / table). A
  code-only agent (random, heuristic, pure search) reports the bytes of any data blob it loads (0 if
  none). Memorized tables pay full price here.
- **params** ‚ÄĒ number of learned scalar parameters (0 for search/heuristic).
- **flops_per_move** ‚ÄĒ declared arithmetic ops for one `select_move` at the position's branching. The
  bench **verifies plausibility**: for an `nn` agent, `flops ‚Čą 2 √ó params √ó (forward passes)` within a
  tolerance; a declared value below a floor or above the measured-latency-implied ceiling is rejected.
- **latency_ms** ‚ÄĒ measured p50 wall-time of `select_move` over `sealed` (JIT/warmup excluded).

## 4. Compute ceiling (anti-cheat gate)

An agent whose measured `latency_ms` p50 exceeds **`LATENCY_CAP_MS = 50`** (per move, on the reference
sandbox) OR whose `flops_per_move` exceeds **`FLOP_CAP = 5_000_000`** is marked `over_budget = true`.
Over-budget agents still appear on the board and can win the *unbounded* tier, but are **excluded from
the cheap budget tiers** and from the headline prize. (This is what stops "just embed a full solver /
huge table and call it cheap".)

## 5. Budget tiers (the "holes" of the golf course)

By `size_bytes`:

| Tier   | size_bytes ‚Č§ | intent                                  |
|--------|--------------|-----------------------------------------|
| nano   | 4 096        | 4 KB ‚ÄĒ extreme compression              |
| micro  | 32 768       | 32 KB ‚ÄĒ the **headline** budget         |
| mini   | 262 144      | 256 KB                                  |
| small  | 2 097 152    | 2 MB                                    |
| open   | ‚ąě            | unbounded (perfect solver lives here)   |

An agent qualifies for a tier iff `size_bytes ‚Č§ tier.cap` **and** not `over_budget` (except `open`,
which ignores the compute ceiling). Within a tier, rank by `optimality` desc, then `soundness` desc,
then `latency_ms` asc, then `size_bytes` asc.

## 6. `neurogolf_score` ‚ÄĒ the single scalar (leaderboard rank key)

```
strength          = optimality                       # in [0,1]
size_kb           = size_bytes / 1024
efficiency_pen    = log2(1 + size_kb)                 # 0 at 0KB, grows slowly
soundness         = 1 - blunder_rate
neurogolf_score   = 100 * (0.85 * strength + 0.15 * soundness) / (1 + 0.15 * efficiency_pen)
```

Higher is better. A tiny, near-optimal net beats both a big perfect table (huge `efficiency_pen`) and
a cheap-but-weak agent (low `strength`). Over-budget agents get `neurogolf_score` computed but flagged;
they cannot hold a cheap-tier crown. Round to 3 decimals; ties broken by the tier ordering in ¬ß5.

## 7. Pareto frontier (the true picture)

On the plane **x = cost, y = strength** an agent `A` **dominates** `B` iff
`A.optimality ‚Č• B.optimality` and `A.size_bytes ‚Č§ B.size_bytes` and `A.flops_per_move ‚Č§ B.flops_per_move`,
with at least one strict. The **frontier** = the non-dominated set. `leaderboard.json` marks each agent
`pareto: true|false` and emits the ordered frontier points for the UI plot (both size and FLOPs axes).

## 8. Headline gate metric (what "target met" means)

```
HEADLINE = max optimality over agents that qualify for the `micro` tier (‚Č§32KB, not over_budget)
```

This is the project's headline number. The flagship
`neurofour-net` should own it. A secondary published number is the frontier's **area under the
strength‚Äďlog(size) curve** (AUC), rewarding an agent that is Pareto-efficient across *all* tiers.

## 9. Determinism & reproducibility (the bench must guarantee)

- Same seed + same agents ‚áí byte-identical `leaderboard.json` (except `latency_ms`, which is reported
  but never used in the pass/fail gate ‚ÄĒ only in intra-tier tiebreaks, and there rounded to a stable
  bucket).
- `scripts/run_bench.py --check` recomputes and asserts the committed `leaderboard.json` matches
  (modulo latency) ‚ÄĒ used by `Verify:`.
- Re-generating positions with a *different* seed and re-scoring must not materially change a
  well-generalizing agent's optimality (guards against test-set memorization).
